## 10. RT-5 audit followup: cascade-guard sweep

**Origin.** Post-M1-archive suggestion surfaced during the M1 closure review (after commit `1342321` flipped the verdict from `gaps_found` to `passed`): *"Open a quick followup to grep `apps/` for any other unguarded Pydantic constructions that share the same cascade-crash bug class as `44f8b9d` fixed."* This section documents the resolved sweep.

**Discovered sites.** 3 unguarded Pydantic-construction sites across `apps/`. Each has a distinct blast radius + risk profile; per-site verdict graded below.

### Site 1: `apps/dlq_consumer/worker.py` `_emit_feed_unhealthy` -- **GUARDED**

The `reason` field is constructed as `f"DLQ message for {event_type}"`. `event_type` is derived from `body.get("event") or original_routing_key` (potentially user-controlled routing keys from `x-death` headers) AND from `f"dlq.depth.critical:depth={depth}"` from the depth-probe path (the depth integer is broker-reported and unbounded). Without an inner guard, a >120-char `event_type` raises `ValidationError` from `FeedUnhealthy(reason=...)`, propagates out of `_handle_message` -> `_on_message` -> into `async with message.process()`'s exception exit, which nacks the DLQ message (aio_pika default `requeue=False`). Net effect: the bad message keeps re-entering DLQ in an unbounded nack-cycle. The DLQ consumer process stays alive, but the cycle is operationally invisible -- a single malformed upstream routing key could create an infinite poison-message loop.

**Resolution: commit `07f54d5`.** Inner `try/except ValidationError` wrapper around `payload = FeedUnhealthy(...)` inside `_emit_feed_unhealthy`. On `ValidationError`: `log.error("Discarding feed.unhealthy event for event_type=%s due to schema validation failure (reason > 120 chars? malformed ts?): %s", event_type, e)` + early `return`. Mirror of the `44f8b9d` Option B pattern. Regression test `tests/test_dlq_consumer.py::test_emit_feed_unhealthy_skips_validation_error` (async via `@pytest.mark.asyncio` + `await`) drives `_emit_feed_unhealthy(event_type="dlq.depth.critical:depth=" + "9"*200)` (226 chars exceeds `Field(max_length=120)`) and asserts: function does NOT raise, `_events_exchange.publish` is NEVER called (log-and-skip), 1+ `Discarding feed.unhealthy event` ERROR-severity log line was emitted. Code-reviewer PASS on both source + the 2-token test fix.

### Site 2: `apps/brief/consumer.py` `process_verdict` -- **SAFE-AS-IS**

`SabPublished(event=..., pub_ts=..., snapshot_day=..., ccir_topics=..., bluf_by_topic=..., item_refs=..., total_keep=...)` is constructed inside `process_verdict` but the `process_verdict` call is wrapped in an outer `try/except Exception as e: log.error(...); raise` inside `on_verdict_message`. A `ValidationError` raised inside the `SabPublished(...)` construction is caught -> logged at ERROR with item_id + `exc_info=True` -> re-raised -> `message.process()` nacks the bad `verdict.ready` to DLQ for inspection. The consumer process does NOT crash; the bad message is durably persisted in DLQ + the operator gets a stack trace in logs. **Acceptable behavior; no change required.**

### Site 3: `apps/triage/worker.py` `process_item` -- **SAFE-AS-IS**

`VerdictReady(...)` is constructed with values that always pass Pydantic validation:

- `event="verdict.ready"` -- hardcoded Literal.
- `item_id` -- `str`, no constraint.
- `ccir` -- `Optional[str]` from `fields.get("ccir")`, no constraint.
- `cnr = map_cnr(fields.get("cnr") or "none")` -- mapper is total: `"none" -> "Routine"`, `"I" -> "I"`, `"II" -> "II"`.
- `score = clamp_score(fields.get("score", 0))` -- int in [0, 10] forced via `clamp_score(value, 0, 10)` helper.
- `bucket = map_bucket(fields.get("bucket") or "skip")` -- mapper is total: `"read" -> "keep"`, `"maybe" -> "maybe"`, `"skip" -> "skip"`.
- `why = fields.get("why") or ""` -- `str`, no max_length on schema.
- `ts = datetime.datetime.now(tz=datetime.timezone.utc)` -- tz-aware UTC, satisfies `AwareDatetime`.

Pydantic validation cannot fail in practice. **Cascade-crash impossible; no change required.**
