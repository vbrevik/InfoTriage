## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Shared dev/test Postgres | `DEV_DSN` (used by test fixtures across the repo) is the SAME live Postgres the running docker-compose stack uses — not an isolated test DB. Any test file with a live-DB-touching fixture can wipe or pollute production data. Two files had this exact footgun this session (fixed), but the underlying architecture (shared DSN) is unchanged. | advisory | Before running any test file against Postgres, `grep` it for `TRUNCATE`/`_truncate_all`/fixture-level `psycopg.connect` and confirm the postgres-param variant actually carries `pytest.mark.db_live` (not just a same-named `skipif`-only helper) before trusting `-m "not db_live"` to protect you. |
| Live-consumer test contention | `tests/test_bus_rabbitmq.py` and `tests/test_bus_consume.py` share `q.triage` with the real production `infotriage-triage` worker when both run at the same time — the worker steals test messages, causing false test failures (not real regressions). | advisory | Stop the `triage` container before running RabbitMQ-consumer-touching tests, restart after. Confirmed via stop→pass, restart→fail reproduction this session. |

Noting it here only so it isn't re-investigated from scratch. | advisory | If ingest silently does nothing again, don't re-chase this angle — check Postgres connection health / contention instead (that's what actually happened last time, most likely). |
