  <behavior>
    - put_blob(root, b"hello") returns sha256 hex of b"hello"; the file exists at root/<h[:2]>/<h[2:4]>/<h>.
    - get_blob(root, h) returns the exact original bytes.
    - Calling put_blob twice with identical bytes creates exactly one file and raises no error (dedup no-op).
    - Distinct byte payloads produce distinct hashes and distinct sharded paths.
    - get_blob(root, "../../etc/passwd") raises ValueError (not a 64-hex hash) before any filesystem access.
    - get_blob on a valid-format-but-absent hash raises FileNotFoundError.
    - An induced write failure (e.g. patched write_bytes raising) leaves no file at the final path and re-raises (no silent success).
  </behavior>
  <action>
    Author `libs/store/src/store/_blob.py` as a stdlib-only module (hashlib, os, re, pathlib) exposing module-level functions `put_blob(root: Path, data: bytes) -> str`, `get_blob(root: Path, blob_hash: str) -> bytes`, and helpers `_shard_path(root, h) -> root/h[:2]/h[2:4]/h` and `_validate_hash(h)`.

    put_blob: compute `hashlib.sha256(data).hexdigest()`; if the sharded dest already exists, return the hash (dedup no-op, R4 idempotency); else mkdir parents, write to a sibling `.tmp` file, then `os.replace(tmp, dest)` for atomicity (POSIX rename; never os.rename — Anti-Patterns). On any exception, `unlink(missing_ok=True)` the temp file and re-raise — never leave a partial write and never swallow (fail-loud, must-NOT no-silent-data-loss).

    get_blob: call `_validate_hash` first — `re.fullmatch(r'[0-9a-f]{64}', blob_hash)` or raise ValueError (T-02-02 path-traversal guard, V5 input validation). Then read and return the sharded path's bytes (FileNotFoundError propagates on miss).

    Author `tests/test_store_blob.py` (plain pytest, `tmp_path` fixture) covering every bullet in <behavior>, including the negative atomicity test (monkeypatch the temp write to raise and assert no final-path file) and the traversal-guard test.
  </action>

  </verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → blob store | `blob_hash` supplied to get_blob is untrusted and is used to build a filesystem path |
| caller → store | `Item` content (title/url/payload) crosses into persistence; payload is open dict |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-02 | Tampering | _blob.get_blob path construction | high | mitigate | `_validate_hash` rejects any blob_hash not matching `^[0-9a-f]{64}$` before constructing a path — blocks `../` traversal; hash is content-derived in put_blob, never from untrusted external input (V5 input validation). |
| T-02-04 | Tampering | Item.payload deserialization | medium | accept | Phase 2 stores/returns payload as an opaque dict; no eval/pickle. JSONB safety + secrets review are owned by PostgresStore (plan 03, Jsonb wrapper) and /gsd-secure-phase. No bespoke control here. |
| T-02-05 | Denial of Service | unbounded list_items / render_atom | low | mitigate | `list_items` and `render_atom` carry a `limit` (default 200 / 50) so a large store cannot force unbounded result materialization. |
</threat_model>
