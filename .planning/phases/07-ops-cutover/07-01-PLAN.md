# 07-01-PLAN.md — M1 ship-gate ops

## Goal

Make the InfoTriage stack operable for M1 ship-gate by completing four
focused work streams. Each task is independently shippable; the order
below is the recommended execution order (smallest/cheapest first).

1. Retire legacy host-run scripts (fever_triage.py, gmail_to_atom.py)
2. Structured container logging end-to-end (json-log-formatter partly wired in commit edbf283)
3. ops/Makefile for the operator
4. Dead-letter queue (DLQ) consumer

This plan supersedes the original 5-plan Phase 7 structure (07-01 FreshRSS
ops is already done; 07-02/07-03/07-04 are rolled up here; 07-05 M1
verification will become the plan's own verification section).

### Task 2: Structured container logging end-to-end

- Confirm `json-log-formatter` is wired into all 8 services (ingest-imap, ingest-youtube, ingest-gmail, ingest-obsidian, triage, brief, opml-health, scheduler) — commit edbf283 ("Phase 6 UAT cold-start pass, opml-health service, and NewsAPI TTL ops" milestone) wired it into ingest/triage/brief/opml-health; extend to the rest
- Configure `LOG_LEVEL` env var per service (default INFO; DEBUG via .env for triage while developing)
- Add log file rotation: `data/logs/<service>.log`, daily rotation, 7-day retention
- Add a `jq` query cookbook in `docs/ops/logging.md` (filter by level, by service, by event type)
- Acceptance: all 8 services emit JSON to stdout; logs land in `data/logs/<service>.log`; `docker compose logs brief | jq -r '.message'` returns parseable JSON
- Verification: trigger a verdict.ready via `scripts/uat_test7_event.py`; grep the brief container logs for the JSON `published SabPublished` event

### Task 3: ops/Makefile

- `make up` — `docker compose up -d`
- `make down` — `docker compose down`
- `make logs` — `docker compose logs -f` with optional `make logs-<service>` shorthand
- `make status` — `docker compose ps` + health-check (curl each service's /health)
- `make restart` — `docker compose restart`
- `make shell-<service>` — `docker compose exec <service> bash`
- `make seed` — `python3 scripts/seed_sample_data.py`
- `make backfill` — re-runs ingest-imap + ingest-youtube + ingest-gmail + ingest-obsidian
- `make replay` — drain + replay DLQ (delegates to the DLQ consumer's replay subcommand)
- `make help` — auto-generated from comments above each target
- Acceptance: all 9 targets work without error; `make help` is self-documenting; targets are idempotent (running `make up` twice is safe)
- Verification: each target smoke-tested from a clean `docker compose down` state

### Task 4: Dead-letter queue (DLQ) consumer

- Add a small worker (likely in `apps/opml_health/` or a new `apps/dlq_consumer/`) that subscribes to `q.ops.dlx`
- For each DLQ message: log at ERROR level + emit `feed.unhealthy` event if the message is a known event type
- Implement bounded depth: after N consecutive errors, page the operator (log at CRITICAL; future: ntfy push per Phase 12)
- Add a `make replay` subcommand that re-publishes N messages from the DLQ back to their original exchange/routing-key
- Wire the consumer into `docker-compose.yml` (port 22033 for health)
- Acceptance: publish a poisoned verdict.ready to `infotriage.events`; verify it dead-letters; verify the DLQ consumer logs the message and emits `feed.unhealthy`; verify `make replay` recovers it
- Verification: current `q.ops.dlx` has 30 messages backlog — drain them via the new consumer and confirm each gets logged
