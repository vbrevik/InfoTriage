# Plan 06-02: pgvector semantic clustering (R4)

**Requirement:** spec §Reading-surface routing — R4
**Depends on:** 06-01 (consumer, markdown renderer, HTML server, SabPublished event)
**Scope:** Replace keyword-overlap fallback in `renderer.py` with pgvector HNSW semantic clustering. Create `clustering.py`, modify `renderer.py`, add tests. No changes to consumer.py or Dockerfile.

**Why keyword-overlap is wrong:** The current `digest.py` `cluster()` uses greedy keyword-overlap (2+ shared 4+ char tokens). This incorrectly merges semantically similar but lexically distinct articles (e.g., "NATO Ukraine defense spending" + "NATO Arctic policy" — both share "NATO" but cover different topics). pgvector cosine similarity on mE5-large embeddings correctly separates them.

**Acceptance criterion:** Given 3 articles about "NATO" (2 Ukraine defense, 1 Arctic policy) with embeddings, pgvector clustering with threshold 0.75 merges only the 2 Ukraine articles — the Arctic article remains a separate single-item cluster.

### 1. Create `apps/brief/clustering.py` — pgvector HNSW per-CCIR clustering

**Read first:**
- `libs/store/src/store/_postgres.py` — PostgresStore class, `find_near_duplicate()` SQL pattern, `register_vector()` call in `__enter__`
- `libs/store/src/store/_protocol.py` — Store Protocol, `find_near_duplicate()` signature
- `apps/triage/digest.py` — `cluster()` function (keyword-overlap fallback, to be replaced)
- `.planning/phases/06-brief-app/06-SPEC.md` — R4 acceptance criterion (NATO/Ukraine/Arctic example)

**Action:**
Create `apps/brief/clustering.py` with function `cluster_items(store, ccir_sections, threshold, window_hours)`. For each CCIR section in `ccir_sections`:
1. Query `infotriage.enrichment JOIN infotriage.articles ON enrichment.item_id = articles.id JOIN infotriage.embeddings ON enrichment.item_id = embeddings.item_id WHERE enrichment.ccir = %s AND enrichment.created_at >= %s ORDER BY enrichment.created_at DESC`
2. For each pair of items in the same CCIR section, compute cosine similarity via pgvector: `SELECT 1 - (embedding <=> %s::vector) AS similarity FROM infotriage.embeddings WHERE item_id = %s`
3. Greedy assignment: for each item (descending by score), find nearest neighbor above threshold within the same CCIR section. If found and its cluster hasn't reached capacity (or no capacity limit), add to that cluster. Otherwise start a new single-item cluster.
4. Return `list[list[enriched_item]]` — each inner list is a cluster (at least 1 item).

Threshold default: 0.75. Configurable via `CLUSTER_THRESHOLD` env var in `main.py` and passed into `cluster_items()`.

**Acceptance criteria:**
- `apps/brief/clustering.py` contains `def cluster_items(`
- SQL queries use `%s` bind params only (no f-strings)
- `clustering.py` imports `register_vector` check: `register_vector()` already called by `PostgresStore.__enter__` — no additional call needed
- Function returns `list[list[dict]]` where each inner dict has keys: `item_id`, `title`, `source`, `url`, `summary`, `ccir`, `cnr`, `score`, `bucket`, `why`, `pmesii`, `tessoc`, `embedding_similarity`
- `test_clustering.py` has at least 3 tests: single cluster, multiple clusters, CCIR boundary enforcement
