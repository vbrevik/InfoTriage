<diagnosis>
Root cause (verified live in-container during UAT, and re-confirmed against source here):

`write_vault_digest()` (apps/brief/vault_writer.py, ~line 252-254) excludes email with:
`kept = [r for r in kept if not (r.get("source") or "").startswith("imap://")]`

This tests the wrong field. The `source` column holds the adapter/mailbox name, never an `imap://` URI. The mail URI lives in `url`. Verified field values emitted by the three email-capable adapters:

- ingest-imap  (apps/ingest-imap/imap_ingest.py:166-168): source=<mailbox name> (e.g. "Telegraph Ukraine"), source_type="imap", url=`imap://<host>/<message_id>`
- ingest-gmail (apps/ingest-gmail/gmail_ingest.py:103-105): source="gmail",              source_type="gmail", url=`gmail://message/<msg_id>`
- ingest-obsidian (apps/ingest-obsidian/obsidian_ingest.py:115-117): web-clip, source_type="obsidian" — NOT email, must NOT be excluded.

Two consequences the executor must respect:

1. The current predicate excludes NEITHER production shape: gmail's source="gmail" and imap's source=<mailbox name> both fail `.startswith("imap://")`. Only the synthetic control shape (source="imap://…") used in the UAT was ever caught.
2. A naive one-line fix to `url.startswith("imap://")` would STILL miss gmail — the actual production leaker — because gmail rows carry `url=gmail://message/…`, not `imap://`. The predicate MUST cover both email URL schemes.

Why url-scheme, not source_type: `source_type` ("imap"/"gmail"/…) is the canonical column but the brief consumer's SELECT (apps/brief/consumer.py:61-68 selects `a.source, a.url` only, not `a.source_type`), so source_type does not reach these rows. A url-scheme check keeps the fix entirely inside vault_writer.py — surgical, no consumer/SQL change.
</diagnosis>
