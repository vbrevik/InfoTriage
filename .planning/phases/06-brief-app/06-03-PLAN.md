## Objective

Fix the root cause of SC1b: production data path never joins `infotriage.embeddings`, so every item gets `embedding=[0.0,0.0,0.0,0.0]`, always breaking the threshold and never merging.

Two approaches are possible: (A) modify the enrichment queries in consumer.py/main.py to join embeddings, or (B) use `clustering.py`'s pgvector-backed `cluster_items()` function. This plan documents both approaches for review.

## Context

VERIFICATION.md reported:
- `apps/brief/rendering.py`'s `_rows_to_enriched_items()` defaults embedding to `[0.0]*4`
- `cluster_items()` is dead code — never called by consumer.py, main.py, or renderer.py
- All enrichment rows reach `renderer.py` without an embedding column

The fix must ensure real embeddings flow through to clustering.

## Approach Options

### Option A: Join infotriage.embeddings in enrichment queries

Modify `_SELECT` in `consumer.py` and `_ENRICHMENT_SQL` in `main.py` to include:
```sql
:e.embedding
```
via a LEFT JOIN to `infotriage.embeddings`. Then the existing `render_cluster()` code will receive real embeddings.

**Pros:** Simple change, minimal new dependency; fits existing code structure.  
**Cons:** Still uses `cluster_items_in_memory()` in renderer.py, which is fine but not the pgvector SQL path.

### Option B: Call `clustering.py`'s pgvector `cluster_items()` directly

Create a new function in `consumer.py`/`main.py` that calls `cluster_items()` from `clustering.py`, which performs its own SQL query with pgvector cosine similarity. Then `renderer.py` gets clusters directly.

**Pros:** Uses the full pgvector SQL path; cleaner separation of concerns; more efficient (clustering done in DB).  
**Cons:** Requires larger refactor; more architectural change; may need to adjust routing of clusters.

**Decision:** Option A is chosen for simplicity and lower risk, with a clear path to move to Option B later. Option B will be deferred to a future plan.
