- [ ] CONSTRAINT: DO NOT run `pytest` (bare) or `pytest -m db_live` before shadow_run.py's parity check completes ‚Äî `tests/test_store_contract.py`, `test_store_integration.py`, and `test_triage_enrichment.py` all call `_truncate_all(DEV_DSN)` unconditionally, and `DEV_DSN` (`postgresql://infotriage:infotriage_dev@localhost:22000/infotriage`) is the SAME live Postgres the running docker-compose stack uses ‚Äî not an isolated test database. ## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Shared dev/test Postgres | 3 test files truncate the live dev DB as an unconditional fixture side effect (see constraint above) | blocking | Never run those files' `db_live`/postgres-param tests against a running dev stack you care about. Real fix (not done this session, deferred): point `DEV_DSN` at a genuinely separate test DB. |
| AMQP header vs body confusion | `RabbitMQBus.publish()` puts `item_id` in the AMQP message **headers** (`headers={"routing_key":..., "item_id":...}`), never in the JSON body (body is only `{source, source_type, ts}`). `worker.py`'s `on_message` was reading `body["item_id"]` ‚Äî 100% of messages dead-lettered with `KeyError`. Fixed this session (uncommitted) by reading `message.headers["item_id"]` instead. | advisory | Any future RabbitMQ consumer code in this project must read `message.headers` for `item_id`/`routing_key`, never assume they're in the JSON body. |
| Fire-and-forget trigger endpoints | `ingest-youtube`/`ingest-imap`'s `POST /run` returns `200 {"status":"started"}` immediately via `asyncio.create_task` ‚Äî the real ingest work happens after the response is sent. A `200` tells you nothing about whether ingestion actually succeeded; any exception is only `log.exception()`'d, never surfaced. | advisory | Always verify via container logs or direct row counts after triggering `/run`, never trust the HTTP 200 alone. |
