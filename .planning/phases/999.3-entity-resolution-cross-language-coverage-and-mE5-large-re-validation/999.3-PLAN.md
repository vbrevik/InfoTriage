## Task 3 — Threshold selection + verdict

**Why:** The sweep table yields a recommended T*. We commit to a verdict (GO / PARTIAL / NO-GO) based on
the bars in §SPEC.

**Decision rule:**
- T* = smallest T with `collapse_rate == 1.0` AND `control_overmerge == 0`, if exists.
- Else T* = first T with `collapse_rate >= 0.95` AND `control_overmerge == 0`.
- Else T* = 0.85 with explicit NO-GO sub-verdict (no clean separation).

**Output:** the 999.3-VERDICT.md frontmatter `Verdict: GO|PARTIAL|NO-GO` line + the chosen T* row +
the three acceptance criteria rows (PASS/FAIL) + a one-line "Phase 8 handoff" note stating the
production-recommended value of LINK_THRESHOLD for `apps/triage/entities.py`.

**Done when:** verdict is unambiguous.

## Task 4 — Optional multi-day corpus augmentation from production Postgres

**Why:** Even if the 12-entity default corpus clears all bars, Phase 8 wants to know whether
cross-language merges hold up against the **actual** production corpus of 170 articles spanning
days × 3 languages. Without it the verdict is PARTIAL because of corpus coverage.

**Implementation:**
- Add a `--corpus-from-postgres` flag to the script. When set, export `SELECT title, lang FROM
  infotriage.articles WHERE lang IN ('en','no','ru') AND ts > NOW() - INTERVAL '14 days'` via
  `docker exec infotriage-postgres psql`.
- Run the lightweight `_KNOWN_TOPICS` heuristic from `apps/triage/entities.py` over each title +
  a brief regex pass to surface cross-language entity mentions.
- Build a date-stamped cross-language pair list and rerun the threshold sweep.
- If this confirms the 12-entity T*, escalate PARTIAL → GO. If T* deviates by ≥ 0.02, escalate
  PARTIAL → NO-GO with the new T* in the explicit handoff.

**Done when (optional, but expected):** the verdict `999.3-VERDICT.md` includes a line
`Postgres-corpus sweep confirmed T* = X.XXXX` with the new T* repeated if it differs.
